## 2. 编制目的

本文档用于说明在当前 `webCrawler` 项目基础上，建设“策略管理系统”的完整设计方案，包括业务目标、功能结构、系统架构、数据库设计、接口设计、页面设计、运行机制、调度机制、安全机制及实施建议等内容。

本文档的目标是：

1. 为项目新增“策略管理系统”提供正式设计依据；
2. 统一研发、测试、实施对策略平台建设范围的理解；
3. 明确该系统与现有股票抓取、行情同步、财务同步、前端管理页面之间的衔接方式；
4. 指导后续数据库建模、后端开发、前端页面建设与调度任务实现。

---

## 3. 项目背景

当前 `webCrawler` 项目已具备以下能力：

1. 支持股票基础信息抓取与同步；
2. 支持股票历史价格抓取与落库；
3. 支持公司财务信息同步；
4. 支持公司、板块、概念、关注等数据管理；
5. 支持基于 Freemarker + DataTables 的后台管理页面；
6. 支持基于 Spring `@Scheduled` 的定时调度任务；
7. 支持基于 Echarts 的统计看板展示。

从当前项目能力来看，系统已经具备“策略运行所需的数据基础层”和“后台管理页面基础框架”，但尚缺少以下核心能力：

1. 策略定义与管理；
2. 策略脚本编辑与版本控制；
3. 策略手动运行与定时运行；
4. 策略运行实例跟踪；
5. 策略运行结果、日志、信号、评分等详情展示；
6. 策略运行安全隔离与执行超时控制。

因此，有必要在当前项目之上增加“策略管理系统”，形成一个面向股票筛选、信号生成、推荐分析和后续回测扩展的统一平台。

## 5. 设计原则

### 5.1 贴合现有项目技术栈

系统建设应尽量复用当前项目技术体系：

- 后端：`Spring Boot 2.2.5`、`MyBatis-Plus`、`MySQL`
- 前端：`Freemarker`、`jQuery`、`DataTables`、`AdminLTE`、`Echarts`
- 调度：Spring `@Scheduled`
- 编辑器：复用现有静态资源中的 `CodeMirror`

### 5.2 分阶段建设

优先建设“选股策略管理平台”，再逐步扩展到“回测分析平台”与“准交易平台”。

### 5.3 保持领域隔离

策略系统应作为独立业务域建设，避免将策略逻辑直接混入 `CompanyStockService`、`CompanyPriceRecordService` 等已有服务中。

### 5.4 运行可追溯

每次策略执行必须具备独立的运行实例、日志、参数快照和结果记录。

### 5.5 安全可控

策略执行必须控制运行时长、可调用 API 范围和数据访问边界，避免用户脚本影响系统稳定性。

## 6. 系统建设目标

建设完成后，系统应具备以下目标能力：

1. 支持用户在后台新建、编辑、保存和发布策略；
2. 支持以脚本或规则方式定义股票筛选逻辑；
3. 支持对策略配置参数和股票池输入范围进行管理；
4. 支持手动运行和定时运行；
5. 支持保存每次运行的输入参数、日志、结果与状态；
6. 支持按运行维度查看推荐股票、评分与信号；
7. 支持图形化展示策略运行结果统计；
8. 支持后续平滑扩展为回测与收益分析系统。

## 8. 总体方案设计

### 8.1 总体定位

本系统定位为：

> 面向股票筛选、信号生成、推荐分析和后续回测扩展的策略管理平台。

第一阶段重点建设“策略管理 + 运行管理 + 结果展示”能力，不直接建设实盘交易与复杂回测引擎。

### 8.2 总体架构

系统逻辑架构建议划分为 5 层：

1. 表现层
2. 策略管理层
3. 策略执行层
4. 数据服务层
5. 基础设施层

#### 8.2.1 表现层

用于页面展示和接口开放，包括：

- 策略列表页
- 策略编辑页
- 运行记录页
- 运行详情页
- 图表统计页
- 后端控制器接口

#### 8.2.2 策略管理层

负责策略元数据管理，包括：

- 策略主信息
- 策略分类
- 策略参数
- 策略版本
- 策略状态
- 策略调度配置

#### 8.2.3 策略执行层

负责策略执行相关逻辑，包括：

- 运行上下文构建
- 数据装载
- 脚本/规则执行
- 日志收集
- 执行结果汇总
- 异常捕获和超时控制

#### 8.2.4 数据服务层

负责为策略执行提供统一数据接口，包括：

- 股票基础信息
- 历史价格数据
- 财务数据
- 板块概念数据
- 关注股票数据

#### 8.2.5 基础设施层

负责系统基础能力，包括：

- MySQL 存储
- 定时调度
- 线程池
- 日志系统
- 前端静态资源

## 10. 业务流程设计

### 10.1 策略创建流程

```text
用户进入策略列表页
→ 点击“新建策略”
→ 填写策略基础信息
→ 编写策略脚本/规则
→ 配置参数与股票池
→ 保存草稿
→ 发布为新版本
→ 返回策略列表
```

### 10.2 手动运行流程

```text
用户在策略列表点击“运行”
→ 输入本次运行参数
→ 系统创建运行实例
→ 锁定当前发布版本
→ 构建策略执行上下文
→ 装载股票、价格、财务等数据
→ 执行策略
→ 写入运行日志
→ 落库运行结果
→ 更新运行状态
→ 用户查看运行详情
```

### 10.3 定时运行流程

```text
系统调度任务定时扫描可运行策略
→ 判断策略启用状态与 cron 是否到期
→ 创建运行实例
→ 推送到策略执行线程池
→ 执行完成后落库日志与结果
→ 更新策略最近运行状态
```

### 10.4 运行详情查看流程

```text
用户进入运行记录页
→ 选择某次运行记录
→ 查看运行摘要、参数快照、日志列表、结果列表、图表统计
```

### 11.1.1 功能目标

### 11.1.2 页面元素设计

- 策略编码
- 策略分类
- 状态
- 调度方式
- 更新时间区间

#### 列表字段

- 策略 ID
- 策略名称
- 策略编码
- 分类
- 脚本类型
- 当前版本
- 状态
- 调度方式
- 最近运行状态
- 最近运行时间
- 更新时间

#### 操作项

- 新建
- 编辑
- 复制
- 发布
- 运行
- 查看记录
- 启用
- 停用
- 删除

### 11.1.3 交互规则

1. 策略状态为“停用”时，不允许定时调度触发；
2. 没有发布版本的策略，不允许正式运行；
3. 删除策略采用逻辑删除；
4. 当前存在运行中的策略，允许继续编辑，但编辑内容仅保存为草稿，不影响正在执行的版本。

## 11.2 策略编辑管理

### 11.2.1 功能目标

为用户提供策略脚本和参数的编辑能力，并形成可版本化、可发布、可测试的策略定义。

### 11.2.2 页面布局建议

采用左右布局：

- 左侧：基础信息与参数配置区
- 右侧：脚本编辑器区
- 下方：测试运行结果区

### 11.2.3 编辑内容设计

#### 基础信息

- 策略名称
- 策略编码
- 策略分类
- 策略描述
- 备注

#### 脚本编辑内容

- 脚本内容
- 脚本类型
- 语法高亮
- 行号显示
- 保存草稿
- 脚本校验
- 发布版本

#### 参数配置

- 参数定义 JSON
- 默认参数值
- 参数说明

#### 数据范围配置

- 股票池类型：全市场 / 自选股池 / 板块 / 概念 / 指定股票
- 板块过滤条件
- 上市时间限制
- 是否过滤 ST（预留）

### 11.2.4 脚本形式建议

基于当前项目技术栈，第一阶段建议采用以下两种方式之一：

#### 方案一：规则型 DSL

用户以配置化表达式定义规则，例如：

```text
market_cap > 100
and pe < 30
and last_price > 5
order by circulation desc
limit 20
```

#### 方案二：Groovy 脚本

用户以 Groovy 编写脚本，并调用平台提供的限定 API，例如：

```groovy
return stocks()
    .filter(it.totalCapitalization > 100)
    .filter(it.lastPrice > 5)
    .sortDesc(it.lastCirculation)
    .limit(20)
```

### 11.2.5 推荐方案

优先推荐第一阶段采用“规则型 DSL + 固定函数集”的方式，原因如下：

1. 实现成本更低；
2. 语义更可控；
3. 安全风险更低；
4. 更容易落地到当前项目；
5. 便于未来平滑升级到 Groovy 执行。

## 11.3 策略版本管理

### 11.3.1 功能目标

保证策略每次发布都有唯一版本号，并确保运行时绑定的是具体版本，而不是动态变化的编辑态内容。

### 11.3.2 设计规则

1. 保存草稿不增加版本号；
2. 每次发布生成新的版本记录；
3. 每次运行必须绑定发布版本；
4. 历史运行详情可回溯到对应的策略版本；
5. 策略修改后不影响历史运行记录。

### 11.3.3 版本状态说明

- 草稿版本：仅用于编辑态保存
- 发布版本：可用于正式运行
- 历史版本：保留但不可修改

---

## 11.4 策略运行管理

### 11.4.1 功能目标

用于统一管理策略手动运行、定时运行、重跑和异常处理。

### 11.4.2 运行模式

#### 手动运行

由用户点击触发，适用于调试和临时分析。

#### 定时运行

由调度中心根据 cron 表达式自动触发，适用于固定策略扫描。

#### 重跑

针对某一次运行失败或需重新验证，可复制当次参数快照并重新执行。

### 11.4.3 运行状态

建议定义如下状态：

- `PENDING`：待执行
- `RUNNING`：执行中
- `SUCCESS`：执行成功
- `FAIL`：执行失败
- `TIMEOUT`：执行超时
- `CANCELLED`：已取消

### 11.4.4 执行规则

1. 策略运行前必须校验是否存在已发布版本；
2. 每次运行必须生成一条运行实例记录；
3. 每次运行都需要记录参数快照；
4. 同一策略允许运行并发度需可控，建议第一阶段限制为单策略单实例执行；
5. 执行失败必须记录错误信息和异常摘要。

- Top10 推荐股票图表

### 11.6.3 页面交互要求

1. 日志列表支持分页或滚动加载；
2. 结果明细支持排序；
3. 因子详情建议 JSON 展示；
4. 状态为失败时需展示完整异常摘要；
5. 未来可扩展为“收益曲线”和“回撤曲线”。

---

## 11.7 调度管理

### 11.7.1 功能目标

支持策略自动运行，并通过统一调度扫描实现任务触发。

### 11.7.2 调度方案

不建议为每个策略动态注册 Spring `@Scheduled`，建议采用统一调度扫描方案：

```text
系统每分钟执行一次调度扫描任务
→ 查询启用且已配置 cron 的策略
→ 判断 next_trigger_time 是否到期
→ 生成运行实例
→ 投递执行线程池
→ 更新调度时间
```

### 11.7.3 调度优势

1. 易于管理和扩展；
2. 支持数据库动态修改 cron；
3. 支持统一日志与失败处理；
4. 避免大量动态任务对象带来的复杂性。

## 19. 安全设计

## 19.1 脚本安全控制

若引入脚本执行，必须限制脚本能力边界。脚本不得：

1. 访问文件系统；
2. 发起任意网络请求；
3. 创建线程；
4. 直接访问 Spring 容器全部 Bean；
5. 无限循环执行。

## 19.2 开放 API 范围

仅允许调用平台提供的有限接口，例如：

- `stocks()`：获取股票池
- `price(stockCode, days)`：获取价格序列
- `finance(stockCode)`：获取财务指标
- `log.info()`：输出日志
- `result.add()`：添加结果

## 19.3 超时控制

每次执行必须设置超时限制，建议：

- 单次策略运行超时：30 秒
- 超时后状态置为 `TIMEOUT`
- 记录超时日志与异常摘要

## 19.4 并发控制

建议第一阶段约束：

1. 单策略同一时刻仅允许一个运行实例；
2. 全局执行线程池大小需配置化；
3. 调度触发和手动触发共享统一执行队列。

---

## 20. 性能设计

## 20.1 性能目标

1. 策略列表分页查询响应时间小于 1 秒；
2. 运行记录分页查询响应时间小于 1 秒；
3. 单次普通选股策略执行时间控制在 30 秒以内；
4. 运行详情页基础信息加载时间小于 2 秒。

## 20.2 性能措施

1. 策略运行采用线程池异步执行；
2. 运行日志、结果批量插入；
3. 常用股票池与基础数据尽量按批查询；
4. 对运行列表和结果列表建立必要索引；
5. 超大日志分页读取。

---

## 21. 日志与监控设计

## 21.1 业务日志

每次策略运行应单独记录：

- 启动日志
- 参数日志
- 数据装载日志
- 执行过程日志
- 成功/失败日志

## 21.2 系统监控指标

建议后续增加以下监控项：

- 当日运行次数
- 运行成功率
- 平均运行耗时
- 超时次数
- 失败次数
- 单策略近 7 日命中率（预留）

## 22. 异常处理设计

## 22.1 异常分类

1. 参数异常
2. 策略脚本语法异常
3. 数据缺失异常
4. 执行超时异常
5. 系统内部异常

## 22.2 异常处理原则

1. 所有异常都必须落库到运行实例和运行日志；
2. 用户页面必须能看到错误摘要；
3. 不应因单次策略运行异常影响其他策略运行；
4. 调度任务必须具备容错能力。

## 26. 结论

基于当前 `webCrawler` 项目的已有能力，建设“策略管理系统”具备较好的基础条件。项目已经拥有股票、价格、财务、板块等核心数据来源，也已经具备后台页面、统计图表和定时任务等支撑能力。

建议采用“独立策略域 + 统一运行实例 + 统一调度中心 + 统一结果落库”的方案推进建设。第一阶段以“选股/评分/信号型策略平台”为目标最为合适，既可以快速落地，又与当前项目已有结构高度契合。

通过本次设计，系统后续可平滑扩展到更复杂的策略回测、收益分析、模拟持仓甚至交易执行场景。
